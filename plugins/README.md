# Plugins

`plugins/` contains the canonical installable AI IDE plugins for the AI
Engineering Platform. A plugin is the unit users install with commands such as
`ai-engineering install platform security --target cursor`.

Each plugin owns provider-neutral commands, skills, agents, rules, templates,
workflows, schemas, and `plugin.yaml` metadata. Shared policy belongs in
`core/`; provider-specific output is generated by `adapters/`; optional runtime
tooling is copied from `mcp-servers/` only when a plugin needs MCP.

## Plugin Anatomy

Plugin directories follow this shape:

- `plugin.yaml`: canonical metadata, dependencies, assets, commands, skills,
  runtime requirements, and provider compatibility.
- `commands/`: user-facing command contracts that map intent to required skills
  and optional MCP tools.
- `skills/`: executable agent instructions and supporting resources.
- `agents/`: plugin-owned agent definitions when needed.
- `rules/`: plugin-owned rules when needed.
- `templates/`: plugin-owned templates when needed.
- `workflows/`: plugin-owned workflow notes when needed.
- `schemas/`: plugin-owned schemas when needed.

Unused asset groups remain as folders and are declared as `none` in
`plugin.yaml`. Do not add README placeholder files just to keep folders alive.

## Skill Ownership Rules

- Each runtime skill has exactly one canonical owner: the plugin containing
  `skills/<skill>/SKILL.md`.
- `core/routing/skill-registry.yaml` maps canonical skill owners and must match
  skill folders and `plugin.yaml.skills`.
- `plugin.yaml.assets.skills` may include shared skills from another plugin when
  a command needs to install them, but it does not make that plugin the
  canonical owner.
- Put implementation, stack, and source-code skills in `application`; put system
  boundary and design-method skills in `architecture`; put repository-wide
  policy and managed agent baselines in `core/agents`, not in a runtime plugin
  skill.

## Dependency Rules

- Required dependencies are installed before the requested plugin. For example,
  installing `application` also installs `architecture`.
- Optional dependencies are not installed automatically unless explicitly
  requested.
- Shared assets can be owned by multiple plugins; ownership is tracked in the
  target project under `.ai-engineering/ownership.json`.

## Change Checklist

- Update `plugin.yaml` whenever commands, skills, dependencies, adapters, hooks,
  runtime settings, or installable assets change.
- Keep command ids and MCP tool ids namespaced by capability.
- Keep plugin command metadata aligned with `core/routing/command-registry.yaml`
  and `mcp-servers/<plugin>/mcp.json` when MCP is used.
- Update English `README.md` first, then synchronize `README_VI.md`.
- Run `npm run validate` after plugin metadata, command, skill, dependency, or
  MCP mapping changes.
